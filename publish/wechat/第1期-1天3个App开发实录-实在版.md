其实没什么高大上的理由。

就是最近在学习 SwiftUI，发现一个问题：
**看教程的时候觉得懂了，一动手就废。**

所以我想，不如逼自己一把，

选这 3 个 App 的原因：
1. **功能简单** - 都是基础功能，适合练手

3. **技术覆盖** - 能练习到 SwiftUI 的核心用法

## 开发流程

我的开发流程很简单，就 5 步：

```
1. 需求分析 (30 分钟)
2. 产品设计 (30 分钟)
3. 代码开发 (4-5 小时)
4. 测试优化 (1 小时)
5. 文档发布 (30 分钟)
```

每个环节都有具体的方法，下面详细说。

---

## 一、需求分析 (30 分钟)

我的做法很简单，就回答 4 个问题：

### 1. 这个 App 是给谁用的？

例如下午做的 HydroTrack (喝水提醒)：
- **用户**: 上班族、学生
- **场景**: 工作/学习时容易忘记喝水
- **需求**: 定时提醒，简单记录

### 2. 核心功能是什么？

我强制自己：**最多 3 个功能**。

HydroTrack 的 3 个功能：
1. 记录喝水量
2. 定时提醒
3. 查看统计

没了。不加社交，不加分享，不加用户系统。

### 3. 和现有 App 有什么区别？

说实话，这 3 个 App 都没什么创新。
喝水提醒、小费计算器，都烂大街了。

我的做法是：**不追求创新，追求完整**。

能把一个完整的功能做好，比什么都强。

### 4. 怎么变现？

这个我想得比较简单：
- 定价：$0.99 或 $1.99
- 目标：先上架，再优化
- 预期：能覆盖开发时间就行

## 二、产品设计 (30 分钟)

我不用复杂的设计工具，就两招：

### 1. 纸笔画草图

找张纸，画出每个界面的大致布局：

```
┌─────────────┐
│  顶部状态   │
├─────────────┤
│             │
│  核心内容   │
│             │
├─────────────┤
│  操作按钮   │
├─────────────┤
│ 底部导航    │
└─────────────┘
```

不用画得多好看，自己能看懂就行。

### 2. 列出界面清单

一般 5 个界面以内：
1. 首页
2. 列表页
3. 详情页
4. 设置页
5. 关于页

超过 5 个，说明功能太多了，砍。

## 三、代码开发 (4-5 小时)

这是最耗时的部分，但也是收获最大的部分。

### 项目结构

我用统一的项目结构，3 个 App 都一样：

```
App/
├── App/           # 入口文件
├── Models/        # 数据模型
├── Managers/      # 业务逻辑
├── Views/         # 界面视图
└── Resources/     # 资源文件
```

这样做的好处：
- **开发快** - 不用每次都重新组织
- **维护易** - 找代码很方便
- **可复用** - 代码可以直接复制

### 核心代码

#### 1. 数据模型

每个 App 先定义数据模型：

```swift
// HydroTrack 的数据模型
struct WaterEntry: Identifiable, Codable {
    let id: UUID
    let amount: Int  // 毫升
    let date: Date
    let containerType: ContainerType
    
    enum ContainerType: String, Codable {
        case cup = "杯"
        case bottle = "瓶"
        case glass = "杯"
        case custom = "自定义"
    }
}
```

#### 2. 数据管理器

我写了一个通用的数据管理器，3 个 App 都用：

```swift
class DataManager: ObservableObject {
    func save<T: Encodable>(_ data: T, forKey key: String) {
        if let encoded = try? JSONEncoder().encode(data) {
            UserDefaults.standard.set(encoded, forKey: key)
        }
    }
    
    func load<T: Decodable>(_ type: T.Type, forKey key: String) -> T? {
        guard let data = UserDefaults.standard.data(forKey: key),
              let decoded = try? JSONDecoder().decode(T.self, from: data) else {
            return nil
        }
        return decoded
    }
}
```

这样每个 App 就不用重复写存储逻辑了。

#### 3. 界面视图

用 SwiftUI 开发确实快，代码很简洁：

```swift
struct ContentView: View {
    @EnvironmentObject var dataManager: DataManager
    
    var body: some View {
        NavigationView {
            VStack {
                // 内容
            }
            .navigationTitle("标题")
        }
    }
}
```

### 开发顺序

我的开发顺序是固定的：

```
1. 创建项目结构 (15 分钟)
2. 实现数据模型 (30 分钟)
3. 实现管理器 (1 小时)
4. 实现主界面 (1.5 小时)
5. 实现设置页面 (30 分钟)
6. 联调测试 (1 小时)
```

按这个顺序，不容易乱。

## 四、踩坑记录

开发过程中遇到不少问题，记录几个典型的：

### 坑 1: SwiftUI 状态管理

**问题**: 数据更新了，界面没刷新。

**原因**: 没用对状态管理。

**解决**:
- 视图内的状态用 `@State`
- 视图外的状态用 `@EnvironmentObject`
- 不要混用

### 坑 2: UserDefaults 数据类型

**问题**: 存储的数组取出来是空的。

**原因**: 自定义类型没实现 Codable。

**解决**:
```swift
// 一定要实现 Codable
struct WaterEntry: Codable {
    // ...
}
```

### 坑 3: 推送权限

**问题**: 推送不显示。

**原因**: 没请求权限就直接设置推送。

**解决**:
```swift
// 先请求权限
UNUserNotificationCenter.current().requestAuthorization { granted, error in
    // 用户同意后，再设置推送
}
```

## 五、代码复用

3 个 App，4500 多行代码，其实很多是重复的。

### 复用的部分

| 组件 | 复用次数 | 说明 |
|------|----------|------|
| 颜色扩展 | 3 次 | Color(hex:) |
| 数据管理器 | 3 次 | 通用存储逻辑 |
| 设置页面 | 3 次 | 界面几乎一样 |
| 关于页面 | 3 次 | 完全一样 |

### 不复用的部分

| 组件 | 说明 |
|------|------|
| 数据模型 | 每个 App 不同 |
| 主界面 | 业务逻辑不同 |
| 推送内容 | 提醒内容不同 |

我的建议：
- **通用组件要复用** - 节省时间
- **业务逻辑不复用** - 保持清晰

## 七、收获和反思

### 收获

1. **SwiftUI 熟练了**
   - 之前看教程，总觉得差点意思
   - 这次实战，真正理解了状态管理

2. **开发流程清晰了**
   - 知道每个环节该做什么
   - 不会东一榔头西一棒子

3. **代码组织能力提升了**
   - 学会了对项目结构
   - 知道了什么代码该复用

### 反思

1. **太赶了**
   - 21 小时不休息，身体吃不消
   - 下次应该分 3 天，每天一个

2. **文档不够**
   - 代码注释太少
   - README 写得很简陋

3. **测试不充分**
   - 只在模拟器测了
   - 还没上真机

## 八、给你的建议

如果你也想开发自己的 App，我的建议是：

### 1. 从简单的开始

不要一上来就想做微信、做抖音。
做个计算器、做个记事本，挺好的。

### 2. 完成比完美重要

我这 3 个 App 都不完美，有些功能还能优化。
但先完成，再优化。

### 3. 建立自己的代码库

通用的代码封装起来，下次直接用。
时间长了，你的效率会越来越高。

### 4. 多写文档

代码是写给自己看的，文档是写给别人看的。
开源项目，文档很重要。
